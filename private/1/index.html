1. 30초 자기소개

안녕하세요. 웹 기반 업무 시스템과 핀테크 서비스를 개발·유지보수해 온 이무진입니다.

웨인테크놀로지에서는 Spring Boot, JPA, QueryDSL, Vue.js, Oracle 기반의 NICEabc 차세대 구축 프로젝트에 참여해 기존 금융 서비스 로직을 신규 구조로 전환하고 조회 로직과 API 연동을 개발했습니다.

코비젼에서는 그룹웨어, 전자결재, 포털, 로그인, 조직도, 웹메일 같은 사내 업무 시스템을 고객사 요구사항에 맞게 커스터마이징했습니다. Java/Spring/JSP, ASP.NET WebForms, C#, Oracle, MS-SQL 환경에서 기능 개발, Procedure 수정, 서버 이전 점검, 외부 시스템 연동, 운영 반영까지 수행했습니다.

제 강점은 화면만 보는 것이 아니라 업무 데이터, 권한, 설정, 배포 흐름까지 함께 확인해 운영 중인 시스템을 안정적으로 개선하는 점입니다.

2. 1분 자기소개

안녕하세요. 웹 기반 업무 시스템과 핀테크 서비스를 개발·유지보수해 온 이무진입니다.

저는 약 2년 9개월 동안 Spring Boot 기반 금융 서비스 개발과 그룹웨어 운영 개발을 경험했습니다. 웨인테크놀로지에서는 NICEabc 차세대 구축 프로젝트에 참여해 기존 레거시 시스템의 비즈니스 로직을 Spring Boot, JPA, QueryDSL 기반 구조로 전환하는 업무를 수행했습니다. 대출 신청, 대출 내역 조회 같은 기능에서 조회 조건과 상태값이 중요했기 때문에 기존 로직을 분석하고 신규 구조에서도 같은 업무 결과가 나오도록 구현하는 데 집중했습니다.

이후 코비젼에서는 고객사별 그룹웨어, 전자결재, 포털, 로그인, 조직도, 웹메일 기능을 개발하고 유지보수했습니다. ASP.NET WebForms, C#, Java/Spring/JSP, Oracle, MS-SQL 기반 시스템에서 로그인 정책 개선, 포털 웹파트 구성, 전자결재 Legacy 연동, 파일 서버 업로드, 휴가자 접속 제어, 서버 마이그레이션 점검 등을 수행했습니다.

제가 업무할 때 가장 중요하게 보는 부분은 영향 범위 확인입니다. 운영 중인 시스템에서는 화면 하나를 수정해도 DB Procedure, 다국어, 세션, 권한, 기초설정, 배포 방식까지 연결될 수 있기 때문에 요청을 받으면 먼저 전체 흐름을 확인하고, 변경 파일과 SQL, 테스트 조건, 운영 반영 절차를 정리하려고 합니다. 이런 경험을 바탕으로 새로운 도메인에서도 시스템 구조와 업무 흐름을 빠르게 파악해 안정적으로 기여하고 싶습니다.

3. 핵심 경력 요약

코비젼

  • 기간: 2025.02 ~ 2026.04
  • 직무: 웹개발자
  • 핵심 환경: Java, Spring, JSP, MyBatis, ASP.NET WebForms, C#, Oracle, MS-SQL, IIS, JEUS
  • 핵심 도메인: 그룹웨어, 전자결재, 포털, 로그인, 조직도, 웹메일, 외부 연동
  • 면접 포인트: 운영 중인 사내 업무 시스템을 고객사별 요구사항에 맞게 분석, 개발, 배포, 안정화한 경험

말할 문장:

코비젼에서는 고객사별 그룹웨어를 운영하면서 같은 기능처럼 보여도 실제로는 서버 구성, DB, 배포 방식, 권한, 다국어, 결재 양식이 달라 영향 범위가 매번 달랐습니다. 그래서 기능을 수정하기 전 화면, Controller/Service, JSP, SQL map, Procedure, 기초설정, 다국어 값을 함께 확인하는 방식으로 일했습니다.

웨인테크놀로지

  • 기간: 2023.06 ~ 2024.11
  • 직무: 웹개발자
  • 핵심 환경: Java, Spring Boot, Spring JPA, QueryDSL, Vue.js, Oracle, Laravel, PHP, MySQL
  • 핵심 도메인: NICEabc 차세대 구축, 스탁론/P2P 금융 솔루션
  • 면접 포인트: 레거시 금융 서비스를 Spring Boot 기반 구조로 전환하고, QueryDSL 조회 로직과 API 연동을 개발한 경험

말할 문장:

웨인테크놀로지에서는 기존 금융 서비스의 레거시 로직을 분석해 Spring Boot 기반 구조로 옮기는 업무를 했습니다. 금융 서비스는 조회 조건과 상태값이 결과에 직접 영향을 주기 때문에, 기존 기능이 어떤 기준으로 대출 신청과 내역 조회를 처리하는지 먼저 확인하고 신규 구조에서 동일한 업무 결과가 나오도록 구현했습니다.

4. 프로젝트별 말하기 카드

4-1. NICEabc 차세대 구축 및 핀테크 솔루션 개발

면접에서 쓰기 좋은 한 줄:

기존 금융 서비스의 레거시 로직을 Spring Boot, JPA, QueryDSL 기반 구조로 전환하고, 대출 신청/내역 조회 기능과 웹/앱 API 연동 흐름을 개발했습니다.

상황:

  • 기존 금융 서비스 구조를 Spring Boot 기반으로 개편해야 했음
  • 대출 신청, 대출 내역 조회 등 금융 업무 기능의 정확성이 중요했음
  • Vue.js 화면과 Android/iOS 하이브리드 앱이 같은 API 데이터를 사용해야 했음

행동:

  • 기존 비즈니스 로직 분석
  • Spring Boot 신규 구조에 맞게 백엔드 기능 개발
  • Spring JPA와 QueryDSL로 조회 조건 구성
  • Oracle DB 기반 테이블 설계와 데이터 모델링 수행
  • Vue.js 및 하이브리드 앱 API 연동 흐름 확인
  • Laravel/PHP, MySQL 기반 금융 솔루션 유지보수

성과:

  • 레거시 기능을 신규 Spring Boot 구조로 전환하는 작업에 참여
  • 웹과 앱에서 동일 업무 데이터를 사용할 수 있도록 API 응답 구조와 조회 조건 확인
  • 금융 서비스에서 상태값, 조회 조건, 데이터 정확성을 중요하게 보는 습관을 얻음

꼬리질문 대비:

  • QueryDSL을 왜 썼나?
    • 동적 조건이 많은 조회에서 문자열 쿼리보다 타입 안정성과 조합성이 좋고, 조건별 분기를 코드로 관리하기 쉬웠다고 답변
  • 본인이 맡은 범위는?
    • 레거시 분석, 백엔드 기능 개발, QueryDSL 조회 구성, DB 모델링, API 연동 확인이라고 답변
  • 금융 도메인에서 어려웠던 점은?
    • 상태값과 조회 조건 하나가 결과에 영향을 주기 때문에 기존 로직의 의미를 정확히 파악하는 것이 어려웠다고 답변

4-2. AP시스템 비밀번호 정책, 계정 잠금, 변경 유도 팝업 개선

면접에서 쓰기 좋은 한 줄:

로그인 정책 변경 요구를 단순 팝업 수정으로 보지 않고, 계정 상태, 비밀번호 만료, 모바일 알림, 메일 발송, Procedure 분기까지 포함한 전체 로그인 흐름으로 분석해 개선했습니다.

상황:

  • 비밀번호 만료, 계정 잠금, 변경 유도 팝업, 메일 발송, 모바일 알림이 각각 다른 흐름에서 동작
  • 고객사 요청을 반영하려면 로그인 정책 전체 영향 범위를 확인해야 했음
  • 운영 전사 적용 리스크가 있었음

행동:

  • Login.aspx, PasswordChange.aspx, PasswordChangePopup.aspx, ApprovalMain.aspx.cs, SecurityUtils.cs 분석 및 수정
  • System_U_Password_R 등 로그인 체크 Procedure 조정
  • 잠금 계정, 만료 예정 사용자, 모바일 사용자, 테스트 계정 예외 케이스 분기
  • 기초설정값으로 비밀번호 필수 변경일, 계정 잠금 기간, 메일 발송 조건 확인
  • 전사 적용 전 테스트 계정 조건으로 먼저 검증

성과:

  • 계정 상태별 로그인 흐름을 정리
  • PC/모바일에서 비밀번호 변경 팝업과 알림이 올바르게 동작하도록 개선
  • 운영 반영 전 테스트 계정 제한으로 장애 리스크 축소

꼬리질문 대비:

  • 왜 테스트 계정 조건을 먼저 걸었나?
    • 로그인은 전 사용자가 영향을 받는 핵심 흐름이라 전사 적용 전 제한된 계정으로 검증하는 것이 안전했기 때문
  • 어려웠던 점은?
    • 정상 사용자, 잠금 사용자, 만료 예정 사용자, 모바일 사용자, 테스트 계정이 모두 다른 경로를 타서 예외 케이스를 정리하는 것이 중요했음
  • 배운 점은?
    • 운영 시스템에서는 기능 구현 속도보다 영향 범위와 검증 조건을 명확히 하는 것이 중요하다는 점

4-3. 올림푸스 그룹웨어 메인화면 웹파트 및 포털 구성 개선

면접에서 쓰기 좋은 한 줄:

고객사 포털 메인 화면을 공지, 결재, 신규 입사자, 휴가 현황, 배너, CEO 메시지, QMS 문서, 시스템 바로가기 중심으로 재구성하고, 설정값과 Procedure까지 함께 정리했습니다.

상황:

  • 고객사가 여러 업무 정보를 메인 화면에서 바로 확인하길 원함
  • 단순 UI 배치뿐 아니라 메뉴, 링크, 다국어, 기초설정, Procedure가 연결됨

행동:

  • 배너 이미지/링크, 신규 입사자 기준 기간, 휴가신청 Form ID 등 기초설정 추가
  • OKIS, IT업무지원, 회의실 예약 등 내부/외부 시스템 링크 등록
  • 시스템 바로가기 다국어 값 정리
  • 전자결재, 프로필, 공지, 신규 입사자, 휴가 현황, 배너, CEO 메시지, QMS 문서 웹파트 구성
  • 신규 입사자 조회 Procedure와 공지사항 외부 링크 조회 Procedure 추가/수정

성과:

  • 포털 메인 화면을 고객사 업무 동선에 맞게 구성
  • 운영자가 이후 설정값과 웹파트 관리로 화면을 조정할 수 있게 함

꼬리질문 대비:

  • 하드코딩 대신 설정을 쓴 이유는?
    • 고객사 운영자가 이후 배너, 링크, 기간 등을 변경할 수 있어 유지보수 부담이 줄기 때문
  • 화면 개발에서 DB가 왜 필요했나?
    • 신규 입사자, 공지, 휴가 현황처럼 메인 웹파트는 화면 구성뿐 아니라 조회 기준과 권한, 다국어가 연결되기 때문

4-4. 테크윙 조립도 배포 문서 검색 및 파일 서버 업로드 연동

면접에서 쓰기 좋은 한 줄:

전자결재 완료 문서를 검색 테이블, 파일 서버 업로드, 기존 데이터 마이그레이션까지 연결해 운영 데이터로 재사용할 수 있게 만든 작업입니다.

상황:

  • 조립도 배포 문서를 검색하고 결재 완료 문서의 첨부 파일을 파일 서버로 이관해야 했음
  • 문서 종류별 검색 기준, REV 판단, 저장 경로, 파일명 중복 처리가 필요했음

행동:

  • 검색용 테이블 생성
  • 조회 Procedure, Insert Procedure 작성
  • 조립도 배포 문서 검색 페이지 구현
  • ApprovalWebService, ExecuteLegacy에 신규 case 추가
  • 결재양식자동생성/양식프로세스 연동
  • 파일 다운로드 경로를 기초설정으로 분리
  • 모델명/승인일자 기준 저장 경로와 파일명 분기
  • 기존 완료 문서 데이터를 검색 테이블로 이관하는 마이그레이션 쿼리 작성
  • REV 검색 조건 수정

성과:

8. 강하게 준비해야 할 답변 10개

1. 본인 강점

제 강점은 운영 중인 시스템의 영향 범위를 먼저 확인하고 안정적으로 수정하는 점입니다. 화면 하나를 수정하더라도 DB Procedure, 권한, 세션, 기초설정, 다국어, 배포 방식까지 연결될 수 있기 때문에 요청을 받으면 전체 흐름을 확인한 뒤 수정합니다.

2. 가장 자신 있는 프로젝트

테크윙 전자결재 문서 검색 및 파일 서버 업로드 연동입니다. 결재 완료 문서를 검색 테이블, 파일 서버, 기존 완료 문서 마이그레이션까지 연결한 작업이라 전자결재 데이터가 운영에서 어떻게 재사용되는지 깊게 이해할 수 있었습니다.

3. 가장 어려웠던 프로젝트

AP시스템 비밀번호 정책 개선입니다. 로그인은 전체 사용자에게 영향을 주는 흐름이고, 비밀번호 만료, 계정 잠금, 모바일 알림, 메일 발송, 테스트 계정 조건이 모두 다르게 동작했습니다. 그래서 Procedure와 화면, 계정 상태별 분기를 함께 확인하고 테스트 계정 기준으로 먼저 검증했습니다.

4. 왜 운영 시스템 경험이 강점인가?

운영 시스템은 이미 사용자와 업무 규칙이 연결되어 있어 작은 수정도 장애로 이어질 수 있습니다. 저는 고객사별 다른 환경에서 기능 개발, DB 수정, 배포, 로그 확인까지 경험했기 때문에 새로운 시스템에서도 영향 범위를 확인하며 안정적으로 접근할 수 있습니다.

5. QueryDSL 경험

NICEabc 차세대 구축에서 Spring JPA와 QueryDSL 기반 조회 로직을 구성했습니다. 금융 서비스는 조회 조건과 상태값이 다양해 동적 조건을 명확하게 조합할 필요가 있었고, QueryDSL은 타입 안정성과 조건 조합 측면에서 유용했습니다.

6. Procedure 경험

로그인 체크 Procedure, 신규 입사자 조회 Procedure, 공지 외부 링크 조회 Procedure, 전자결재 검색/Insert Procedure를 분석하거나 작성했습니다. 운영 DB에서는 먼저 SELECT로 영향 범위를 확인하고, 조건과 상태값, 중복 여부를 검증한 뒤 반영하려고 합니다.

7. 배포 경험

IIS 환경에서는 변경 파일, App Pool, 가상 디렉터리, 설정값을 확인했고, JEUS 환경에서는 WAR 생성, 백업, 배포 위치, 로그 확인 절차를 정리했습니다. 운영 반영 후에는 테스트 계정이나 특정 조건으로 먼저 동작을 확인했습니다.

8. 서버 이전 경험

Windows Server 2012에서 2022로 이전하는 과정에서 IIS, .NET Framework, App Pool, SSL/TLS, DB 연결, 파일 경로, 기초설정 값을 확인했습니다. 특히 기존 서버명과 IP가 DB나 설정값에 남아 있으면 신규 서버에서 문제가 생길 수 있어 테이블과 컬럼을 검색해 영향 범위를 확인했습니다.

9. 약점

한 가지 기술만 깊게 오래 다뤘다기보다 여러 구조의 운영 시스템을 분석하고 수정한 경험이 많습니다. 그래서 새로운 기술이나 도메인을 만나면 초반에 구조를 파악하는 시간을 의식적으로 확보합니다. 대신 파악한 내용은 변경 파일, 설정값, 테스트 조건으로 정리해 다음 작업에 재사용할 수 있게 합니다.

10. 입사 후 기여

먼저 운영 중인 시스템의 업무 흐름과 데이터 구조를 빠르게 파악하겠습니다. 이후에는 기존 안정성을 해치지 않으면서 필요한 기능을 개선하고, 운영자가 관리하기 쉬운 설정과 반영 기준을 남기는 방식으로 기여하고 싶습니다.

12. 개인 프로젝트 심층 대비 - ChatAppProject-ui 하네스

12-1. 한 줄 정의

ChatAppProject-ui에서는 AI가 코드를 빠르게 생성할수록 요구사항, 진행 상태, 검증 기준이 흩어지는 문제가 생긴다고 보고, Codex와 Claude를 특정 도구에 종속하지 않는 모델 중립 하네스로 분리해 운영했습니다.

12-2. 30초 설명

ChatAppProject-ui는 Spring Boot와 Next.js 기반의 실시간 마을/채팅 프로젝트입니다. 이 프로젝트에서 단순히 AI에게 코드 생성을 맡기는 수준이 아니라, AI가 작업할 때 따라야 할 규칙과 문서 구조, 리뷰 절차를 하네스로 정리했습니다. 기존에는 Claude Code 중심의 CLAUDE.md에 규칙이 몰려 있었는데, Codex를 주력 작업자로 쓰기 시작하면서 공통 규칙을 docs/harness/로 분리했습니다. 핵심은 특정 AI 도구를 바꿔도 프로젝트의 작업 방식, 검증 기준, 문맥 관리 방식이 흔들리지 않게 하는 것이었습니다.

12-3. 1분 설명

ChatAppProject-ui에서 하네스를 만든 이유는 AI 개발의 병목이 코드 작성 자체보다 “무엇을 만들지, 어떤 기준으로 검증할지, 이전 결정과 충돌하지 않는지”로 이동한다고 봤기 때문입니다.

기존에는 Claude Code 중심으로 CLAUDE.md, hooks, skills, agents를 관리했지만, Codex를 주력으로 쓰기 시작하면 특정 도구 설정에 프로젝트 규칙이 묶이는 문제가 있었습니다. 그래서 공통 운영 규칙을 docs/harness/로 분리하고, Codex는 기본 구현자, Claude 자산은 기존 훅과 스킬, 교차 검증 레이어로 보존하는 구조를 잡았습니다.

하네스의 핵심 구성은 모델 중립 SSoT, 메인 AI와 역할형 서브 에이전트, 슬라이딩 윈도우 컨텍스트, Critic Gate, Spec-driven 4층 모델입니다. Issue는 왜 시작하는지, Spec은 무엇을 달성할지, Track은 현재 어디까지 왔는지, Step은 이번 커밋에서 무엇을 만들지로 역할을 나눴습니다. 또 위험한 변경은 Comprehension Gate에서 왜, 대안, 빈틈을 본인 말로 설명하게 해서 AI가 만든 코드를 그대로 받아들이지 않도록 했습니다.

면접에서 강조할 점:

  • AI 도구 사용 경험이 아니라, AI를 통제하기 위한 개발 프로세스 설계 경험으로 말하기
  • “AI가 알아서 했다”가 아니라 “AI 산출물을 spec, track, critic, test로 제어했다”고 말하기
  • 하네스는 생산성 도구이면서 동시에 품질/문맥/검증 관리 장치라고 설명하기

13. 하네스 구성요소별 답변 카드

13-1. 모델 중립 하네스

상황:

  • 기존 규칙이 Claude Code 중심 문서에 몰려 있었음
  • Codex를 주력으로 쓰기 시작하면서 도구별 설정 파일이 프로젝트 규칙의 단일 진실이 되는 문제가 생김

행동:

  • 공통 규칙을 docs/harness/로 분리
  • AGENTS.md, CLAUDE.md는 각 도구의 얇은 진입점으로 두고 공통 규칙을 참조하게 함
  • Markdown을 기본 저장 포맷으로 유지하고, 사람용 요약이 필요할 때만 HTML 보조 뷰를 고려

결과:

  • Codex, Claude, 이후 다른 도구가 들어와도 같은 규칙을 참조할 수 있는 구조가 됨
  • 특정 모델이나 IDE에 프로젝트 운영 방식이 종속되는 것을 줄임

꼬리질문 대비:

  • 왜 모델 중립이 중요한가?
    • AI 도구는 계속 바뀌지만 프로젝트의 아키텍처 규칙, 테스트 기준, 리뷰 기준은 유지되어야 하기 때문
  • 그냥 CLAUDE.md 하나에 두면 안 되나?
    • 한 도구의 설정 파일이 공통 규칙을 독점하면 Codex나 다른 도구가 들어올 때 규칙 중복과 불일치가 생기기 때문
  • 단점은?
    • 문서가 늘어날 수 있어 인덱스와 라우팅 문서를 관리하지 않으면 오히려 컨텍스트 비용이 커짐

13-2. 메인 AI + 역할형 서브 에이전트

상황:

  • 모든 판단을 한 AI에게 맡기면 설계, 구현, 테스트, 보안, 동시성 검토가 섞임
  • 반대로 모든 서브 에이전트를 매번 호출하면 비용과 중복 검토가 커짐

행동:

  • Main Codex Orchestrator가 사용자 소통, 범위 판단, 구현 통합, 최종 보고를 맡음
  • Domain Engineer, Adapter Engineer, Test Engineer, Security Critic, Concurrency Critic, Research Scout 같은 역할을 필요할 때만 사용하도록 정의
  • 반복 절차는 에이전트가 아니라 docs/harness/skills/의 스킬로 분리

결과:

  • 최종 책임은 메인 AI가 갖고, 전문 검토가 필요한 경우에만 역할을 분리하는 구조가 됨
  • 무분별한 병렬 에이전트 호출을 줄이고, 필요한 검증만 선택할 수 있게 됨

꼬리질문 대비:

  • 왜 메인 AI가 최종 책임을 져야 하나?
    • 서브 에이전트 결과는 의견이므로 서로 충돌할 수 있고, 사용자 요구와 전체 변경 범위를 통합 판단하는 주체가 필요하기 때문
  • 서브 에이전트와 스킬의 차이는?
    • 에이전트는 독립 판단 역할이고, 스킬은 반복 절차와 체크리스트라고 설명
  • 예시는?
    • 인증 변경이면 Security Critic, Kafka/outbox/idempotency 변경이면 Concurrency Critic, 새 API면 Adapter Engineer와 Test Engineer를 호출

13-3. 슬라이딩 윈도우 컨텍스트

상황:

  • 문서와 학습노트가 계속 쌓이면 AI가 매번 모든 문서를 읽을 수 없음
  • 과거 문맥을 너무 많이 넣으면 현재 작업 집중도가 떨어짐

행동:

  • 컨텍스트를 현재 윈도우, 인덱스, 규약, 영구 저장소로 분리
  • 기본 윈도우는 직전 완료 작업과 현재 진행 작업 2개로 제한
  • 오래된 learning, review, knowledge 문서는 인덱스에서 필요할 때만 찾아 읽도록 설계

결과:

  • 세션 시작 시 읽어야 할 최소 문서가 명확해짐
  • 컨텍스트 비용을 줄이면서도 과거 결정에 접근할 수 있는 구조가 됨

꼬리질문 대비:

  • 왜 모든 문서를 자동으로 읽지 않나?
    • 토큰 비용과 노이즈가 커지고, 오래된 결정이 현재 판단을 오염시킬 수 있기 때문
  • 오래된 결정이 필요한 경우는?
    • 같은 도메인에서 이전 결정과 충돌하거나, 이미 겪은 장애 패턴이 반복될 때 인덱스를 통해 찾아 읽음
  • 운영 시스템 경험과 연결하면?
    • 운영 개발에서도 모든 로그와 테이블을 다 보는 것이 아니라, 문제 흐름에 맞게 범위를 좁히는 것과 비슷하다고 연결

13-4. Critic Gate

상황:

  • AI가 빠르게 구현할수록 테스트 부재, 보안 회귀, 동시성 누락, 문서 불일치가 생길 수 있음

행동:

  • 구현 중 자체 검증, 커밋 전 Critic, PR 전 요약 리뷰, CI 자동 검증으로 게이트를 나눔
  • 변경 유형에 따라 Security Critic, Concurrency Critic, Test Engineer, Docs Consistency Check를 선택
  • 인증/인가, DB/JPA/마이그레이션, Kafka/outbox/idempotency, 하네스/CI 변경은 더 강한 검토를 요구

결과:

  • AI 생성 코드도 일반 코드보다 더 좁고 강한 검증을 거치게 됨
  • “돌아간다”와 “운영 가능한 변경이다”를 분리해서 확인할 수 있게 됨

꼬리질문 대비:

  • Critic은 코드 리뷰와 뭐가 다른가?
    • 단순 스타일 리뷰가 아니라 구현자가 놓친 아키텍처, 보안, 동시성, 테스트, 문서 정합성 리스크를 의심하는 역할
  • 매번 강한 리뷰를 하면 느리지 않나?
    • 위험도별로 선택한다. 문서만 변경이면 자체 검증, 인증/메시징/DB 변경이면 전문 Critic을 필수로 둔다고 답변
  • 회사 업무와 연결하면?
    • 운영 반영 전 변경 파일, SQL, 테스트 계정, 로그 확인 절차를 정리했던 경험과 같은 기준이라고 연결

13-5. Spec-driven 4층 모델

상황:

  • GitHub Issue, 채팅, 코드 변경이 섞이면 시간이 지난 뒤 왜 시작했는지, 어디까지 했는지, 이번 커밋의 범위가 무엇인지 추적하기 어려움

행동:

  • Issue, Spec, Track, Step을 분리
  • Issue: 왜 시작하는가
  • Spec: 무엇을 달성하는가
  • Track: 지금 어디까지 왔는가
  • Step: 이번 작업 단위에서 무엇을 만드는가

결과:

  • 요구사항, 진행 상태, 커밋 단위가 섞이지 않음
  • 다음 세션이나 다른 작업자가 들어와도 현재 상태와 원래 목표를 재구성하기 쉬움

꼬리질문 대비:

  • 왜 4층까지 필요한가?
    • Issue만 쓰면 닫힌 뒤 맥락이 휘발되고, Track만 쓰면 요구사항과 진행 상태가 섞이며, Step만 쓰면 전체 목표가 사라지기 때문
  • 너무 무거운 프로세스 아닌가?
    • 단발 핫픽스는 mini-spec/mini-track으로 줄이고, 멀티 step 트랙만 정식 문서를 사용해 비용을 조절한다고 답변
  • 실무 회사에서도 쓸 수 있나?
    • 전부 도입하기보다 운영 반영 요청서, 변경 범위, 검증 조건, 배포 기록을 분리하는 방식으로 축소 적용할 수 있다고 답변

13-6. Comprehension Gate

상황:

  • AI가 구현한 코드를 개발자가 제대로 이해하지 못하면 “동작은 하지만 내 코드가 아닌” 상태가 됨
  • 특히 DB 스키마, API, 인증, 메시징, 트랜잭션, 새 의존성 같은 결정은 근거를 설명할 수 있어야 함

행동:

  • 변경을 13개 위험 카테고리로 분류
  • 단순 CRUD/테스트/문서 변경은 Tier A로 침묵
  • 기존 패턴 재현은 Tier B로 얕은 질문
  • DB/API/이벤트/인증/헥사고날 경계/새 기술은 Tier C로 왜, 대안, 빈틈을 묻도록 설계

결과:

  • 위험한 변경 직전에 본인 말로 결정 근거를 설명하게 됨
  • spec의 decisions에 왜, 대안, 빈틈, 재검토 트리거를 미리 적으면 게이트가 자동 통과되어 문서 품질도 높아짐

꼬리질문 대비:

  • 왜 자바 기초 질문은 제외했나?
    • 목적은 코드 리딩 테스트가 아니라 의사결정 검증이기 때문
  • 가장 중요한 카테고리는?
    • 인증/인가, DB 스키마, 이벤트/메시징, 트랜잭션 경계, 헥사고날 경계
  • 면접에서 깊게 물으면?
    • “AI가 만든 변경일수록 왜 이 선택을 했는지, 다른 대안은 무엇인지, 어떤 신호가 오면 재검토할지 설명 가능해야 한다고 봤습니다.”로 답변

14. 하네스 4단계 꼬리질문 시뮬레이션

질문 세트 A - “하네스가 뭔가요?”

1단계 질문: 하네스가 정확히 뭔가요?

답변:

제가 말하는 하네스는 AI가 프로젝트에서 일할 때 참조하는 운영 규칙, 문서 구조, 역할 분리, 검증 절차를 묶은 개발 환경입니다. 단순 프롬프트가 아니라 AGENTS, CLAUDE, docs/harness, specs, handover, critic gate까지 포함한 작업 통제 구조입니다.

2단계 질문: 그게 그냥 문서 정리와 뭐가 다른가요?

답변:

일반 문서 정리는 사람이 읽기 위한 기록에 가깝습니다. 하네스는 AI가 다음 작업을 수행할 때 실제 입력으로 쓰는 규칙입니다. 예를 들어 새 API를 만들면 spec 작성, track 갱신, test, critic gate가 자연스럽게 이어지도록 라우팅을 정해둔 점이 다릅니다.

3단계 질문: 실제로 품질이 좋아졌나요?

답변:

정량 지표를 완전히 수치화하진 못했지만, 요구사항과 진행 상태가 섞이는 문제는 줄었습니다. 특히 Issue, Spec, Track, Step을 나누면서 다음 세션에서 “왜 이걸 했는지”와 “어디까지 했는지”를 재구성하는 비용이 줄었습니다. 또 보안, 동시성, DB 변경처럼 위험한 작업은 Critic Gate로 별도 확인하게 되어 검토 누락을 줄이는 방향으로 개선했습니다.

4단계 질문: 회사 프로젝트에 그대로 적용할 건가요?

답변:

그대로 가져오지는 않을 것 같습니다. 회사에는 이미 이슈 관리, 배포, 코드 리뷰 방식이 있을 것이기 때문에 먼저 기존 프로세스를 따르겠습니다. 다만 제가 배운 기준, 즉 요구사항과 진행 상태를 분리하고 위험 변경은 검증 게이트를 두는 방식은 운영 반영 문서나 배포 체크리스트 형태로 축소해서 적용할 수 있다고 생각합니다.

질문 세트 B - “AI가 다 해준 거 아닌가요?”

1단계 질문: ChatAppProject-ui는 AI 도움을 많이 받은 건가요?

답변:

네, AI를 적극적으로 사용했습니다. 다만 코드 생성을 맡긴 것보다 중요한 부분은 AI가 만든 결과를 어떤 기준으로 검토하고 통제할지 정한 것입니다.

2단계 질문: 그러면 본인이 한 역할은 뭔가요?

답변:

목표와 비목표를 정하고, 어떤 문서를 기준으로 삼을지, 어떤 변경에 어떤 검증을 붙일지 결정했습니다. 예를 들어 하네스에서는 Codex를 기본 구현자로 두되 Claude 자산은 교차 검증 레이어로 남기고, DB/API/인증 같은 위험 변경은 Comprehension Gate와 Critic Gate를 통과하도록 설계했습니다.

3단계 질문: AI 결과를 어떻게 신뢰했나요?

답변:

신뢰한다기보다 검증 가능한 단위로 쪼갰습니다. Spec에서 달성 기준을 정하고, Step을 커밋 단위로 나누고, 테스트와 Critic Gate를 붙였습니다. 또 위험한 결정은 왜, 대안, 빈틈을 설명할 수 있어야 통과하도록 했습니다.

4단계 질문: AI가 틀리면 어떻게 하나요?

답변:

그래서 메인 AI가 최종 책임을 지고, 서브 에이전트 결과를 그대로 합치지 않는 원칙을 뒀습니다. 실제 개발에서도 AI 제안은 초안으로 보고, 코드/테스트/문서/리뷰를 통해 검증한 뒤 반영해야 한다고 생각합니다.

질문 세트 C - “왜 Spec/Track을 나눴나요?”

1단계 질문: Issue, Spec, Track, Step을 왜 나눴나요?

답변:

각각 답하는 질문이 다르기 때문입니다. Issue는 왜 시작하는지, Spec은 무엇을 달성하는지, Track은 지금 어디까지 왔는지, Step은 이번 커밋에서 무엇을 만드는지 답합니다.

2단계 질문: 그냥 이슈에 다 쓰면 안 되나요?

답변:

이슈는 닫히면 흐름에서 멀어지고, 진행 중 변경사항을 계속 담기에는 요구사항과 상태가 섞입니다. Spec은 목표를 안정적으로 남기고, Track은 변하는 진행 상태를 담게 분리하는 편이 다음 세션에서 더 명확했습니다.

3단계 질문: 관리 비용이 크지 않나요?

답변:

그래서 작업 크기에 따라 다르게 적용했습니다. 큰 기능은 정식 spec/track을 쓰고, 단발 핫픽스는 mini-spec/mini-track으로 줄입니다. 모든 작업에 같은 무게를 강제하지 않는 것이 핵심입니다.

4단계 질문: 실무에서 적용한다면?

답변:

회사에서는 기존 Jira나 GitHub Issue 체계가 있을 테니 그대로 바꾸기보다, 요구사항 문서와 배포 진행 문서를 섞지 않는 방식으로 적용할 수 있습니다. 예를 들어 요청 배경, 구현 범위, 검증 조건, 배포 로그를 분리해서 남기는 식입니다.

질문 세트 D - “Comprehension Gate가 왜 필요한가요?”

1단계 질문: Comprehension Gate는 뭔가요?

답변:

위험한 변경을 커밋하기 전에 그 결정의 이유, 대안, 빈틈을 본인 말로 설명하게 하는 게이트입니다.

2단계 질문: 어떤 변경이 위험한 변경인가요?

답변:

DB 스키마, API 설계, 이벤트/메시징, 인증/인가, 트랜잭션 경계, 외부 시스템 호출, 헥사고날 경계, 새 의존성 같은 변경입니다.

3단계 질문: 왜 모든 코드에 적용하지 않나요?

답변:

모든 코드에 적용하면 피로도가 커져 실제로 중요한 결정이 묻힙니다. 단순 CRUD, DTO 매핑, 테스트 코드, 문서 변경은 침묵시키고 선택의 근거가 필요한 곳만 묻는 구조로 설계했습니다.

4단계 질문: 본인이 이해 못 하면 어떻게 하나요?

답변:

모르면 스킵하고 넘어가는 것이 아니라 학습노트 후보로 남기는 흐름을 뒀습니다. 실무에서도 모르는 결정을 억지로 반영하기보다, 어떤 부분을 모르는지 드러내고 확인한 뒤 반영하는 게 안전하다고 생각합니다.

15. 현재 자료 기준 빈틈 점검

충분히 준비된 영역

  • 회사 경력의 핵심 서사: 운영 시스템, 영향 범위, DB/Procedure, 배포 안정성
  • 프로젝트별 STAR 구조: NICEabc, AP시스템, 올림푸스, 테크윙, 태영건설, 한국가스공사
  • 기술 질문 1~2단계: Spring, JPA/QueryDSL, ASP.NET, SQL, 서버 배포
  • 인성 질문: 강점, 약점, 이직 사유, 입사 후 기여

보강된 영역

  • 개인 프로젝트의 AI 하네스 설계 경험
  • AI 사용을 “도구 활용”이 아니라 “작업 통제 시스템 설계”로 설명하는 답변
  • 하네스 관련 3~4단계 꼬리질문 방어
  • Spec-driven, Critic Gate, Comprehension Gate를 실무 운영 경험과 연결하는 논리

아직 말로 연습해야 하는 영역

  • QueryDSL 실제 코드 예시를 하나 말할 수 있어야 함
    • 예: 조건이 있을 때만 where 조건을 추가하는 동적 조회
  • JPA N+1, 트랜잭션, 인덱스 질문은 개념 답변까지만 준비되어 있음
    • 깊게 들어오면 “실무에서 깊은 튜닝까지 주도한 경험은 제한적이지만, 문제 원인과 대응 방향은 알고 있다”로 방어
  • 하네스는 개인 프로젝트이므로 과장 금지
    • “운영에 도입해 조직 전체가 썼다”가 아니라 “개인 프로젝트에서 AI 협업 품질을 높이기 위해 설계/문서화했다”로 말하기
  • 성과 수치가 부족함
    • 정량 성과 대신 “재현 가능성, 컨텍스트 복원 비용 감소, 검토 누락 감소” 같은 정성 성과로 답변
